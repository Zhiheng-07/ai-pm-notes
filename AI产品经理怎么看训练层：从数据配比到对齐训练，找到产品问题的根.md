# AI产品经理怎么看训练层：从数据配比到对齐训练，找到产品问题的根

## 引子：一件事，两条路

2026 年 5 月 19 日，Andrej Karpathy 正式加入 Anthropic，领导预训练研究。

这件事为什么值得 AI PM 注意？不是因为他名气大，而是因为他加入的时机：就在不久前，OpenAI 联合创始人 Ilya Sutskever 在 NeurIPS 上公开论断——"预训练，就我们所知的这种方式，毫无疑问将会终结"。他的逻辑是：互联网上的数据是有限的，算力再翻 100 倍也不会带来质变，下一个突破口在推理时计算（test-time compute）和 Agent。

Karpathy 没有发文章反驳，他用入职给出了答案：押注预训练还没死，而且值得一流研究者再投入一次。

两条路线，都是顶级从业者在用真实决策表态。对 AI PM 来说，这场争论的意义不在于谁对谁错，而在于它暴露了一个关键事实：**预训练层的路线选择，最终会以模型能力变化的形式传导到你的产品上**——可能是 6 个月后某个版本的能力跃升，也可能是某类任务的悄然退化。

你不需要站队，但你需要看懂这件事背后的机制。这篇文章的目的，就是把这个机制讲清楚。

## 一堵单向透明的墙

在进入技术细节之前，先建立一个认知框架。

训练层和产品应用层之间，隔着一堵"单向透明墙"：

- 训练层的决策（数据配比、Tokenizer 设计、对齐方向）会向上传导到产品端，但这条传导路径是隐性的、有延迟的、非线性的
- 产品端感知到的现象（"这个模型的中文成本比另一家高很多"、"工具调用的格式总是歪"、"有些 System Prompt 怎么写都压不住它的某些行为"）往往找不到直接原因
- **墙是单向透明的**：产品端能感知训练层的结果（通过模型表现），但看不清训练层的决策过程

这种不对称是 AI PM 面对的真实困境。你在应用层折腾 Prompt、调 RAG、试 Fine-tuning，有时候效果差强人意——不是因为你方法用错了，而是训练层早就把上限划死了，你的所有努力都在这个上限以内做优化。

**把这堵墙看透，是这篇要做的事情。**

下面放一张表，把"训练层动作→产品端表现→PM的应对方向"这条因果链收敛成可直接使用的判断框架。先看表，后面三个阶段的技术拆解，都是在给这张表里的每一行提供解释：

**▌因果链表：训练层动作 → 产品端表现 → PM 应对方向**

| 训练层环节 | 在你产品里的可能表现 | PM 的应对方向 |
|---|---|---|
| ①数据工程·数据配比调整 | 模型版本升级后某类任务质量回退 | 升级前跑回归测试；合同里写入"能力不回退"条款 |
| ①数据工程·Tokenizer 设计差异 | 中文输出 token 成本比预期偏高 | 选型时把"中文 token 效率"纳入成本测算，而不只看定价 |
| ③对齐训练·SFT 工具调用覆盖度 | 工具调用准确率到达瓶颈，改 Prompt 效果有限 | 判断是否为模型天花板而非 Prompt 问题，再决定是否换模型 |
| ③对齐训练·RLHF 偏好倾向 | 模型回答风格难以通过 System Prompt 完全调整 | 识别哪些是"权重级偏好"，作为选型依据而非调试对象 |
| ③对齐训练·安全红线 | 部分场景下 System Prompt 写了等于没写 | 区分"Prompt 可改"与"训练层焊死"，避免无效调试 |

记住这张表的逻辑：**产品层的问题，有一部分根源不在产品层，而是训练层写进去的**。知道这件事，你就能把有限的资源用对地方。

# 从语料到对话模型：训练三阶段拆解

## ① 数据工程：模型能力的配方阶段

数据工程是训练三阶段的最上游，也是最容易被忽视的一个——因为它的产出物（训练语料）不像模型那样可以直接测试，它的影响要等到模型训练完才会显现。

但这个阶段的决策，在很大程度上决定了后面所有阶段的天花板。一句话概括这个阶段的本质：**垃圾进，垃圾出——这不是比喻，是字面意义上的因果关系。**

### 语料采集与清洗

模型公司的第一步是从互联网、书籍、代码库、学术论文等来源采集原始语料。原始数据量通常是最终训练语料的数十倍——因为大量内容在清洗阶段会被过滤掉。

清洗做三件主要的事：

**去重**：同一段文字在互联网上可能以微小变体出现数千次。如果不去重，模型会对这些内容过度拟合，变成一个会背诵特定文本的复读机，而不是真正理解内容。去重包括精确去重（完全相同的内容）和模糊去重（高度相似但不完全一致的内容），后者的计算成本更高，但对质量的提升更显著。

**去毒**：互联网上存在大量仇恨言论、欺骗性内容、低质量灌水文本。这些内容进入训练数据，会让模型学会生成有害内容，也会降低模型在正常任务上的表现。去毒通常结合规则过滤（关键词黑名单）和分类模型打分两种方式。但这里有一个精度权衡：过度激进的去毒会把正常讨论敏感话题的高质量文本也过滤掉，反而损害模型的理解能力。

**质量过滤**：并非所有"干净"的文本都是高质量训练数据。模型公司会用多维度指标评估文本质量——语言流畅度、信息密度、逻辑连贯性、原创性等——把低质量内容过滤出去，或者降低其在训练数据中的权重。

清洗之后，Epoch AI 的研究显示，高质量公开文本数据正在接近耗尽——估算区间是 2026 到 2032 年之间。这也是为什么合成数据（由模型生成、再经人工或更强模型审核的训练数据）从补充手段变成了必选项。2026 年 3 月成立的 NVIDIA Nemotron 联盟，其核心工作之一就是联合多家 AI 实验室构建高质量合成数据生态。当前的工业实践是：预训练语料混合中保留 60-70% 的真实人类数据作为锚点，合成数据主要用于"可验证领域的定向放大"——比如数学推理、代码、科学问题，这些领域的合成数据质量更容易用程序化方式验证。

### Tokenization：模型的"视觉单元"

人类读文字，但模型处理的不是字符，而是 token——Tokenization（分词）就是把文字转换成 token 序列的过程。

理解 Tokenization，有两个关键点对 PM 最有实用价值：

**第一，不同语言的 token 效率差异很大。** 英文的分词通常基于词根，"running"可能是一个 token。中文的分词粒度则因 Tokenizer 设计而差异显著——有些模型对中文的编码效率很低，一个中文字符可能对应多个 token，而同样语义的英文内容只需要更少的 token。

这个差异直接影响成本：你的产品如果主要处理中文，选了一个中文 token 效率低的模型，你实际支付的 API 成本会比预期高出 20%-50%，这笔钱不是因为你用了更多功能，而是因为 Tokenizer 设计得不够高效。这也正是因果链表「①数据工程·Tokenizer 设计差异」那行所指的问题：选型时"中文 token 效率"是一个真实的成本变量，不能只看官网的定价页。

**第二，Tokenizer 的切分方式，直接影响模型对同一段文本的理解。** 比如，如果一个 Tokenizer 把"2024-01-15"切成了["2024", "-", "01", "-", "15"] 这 5 个 token，模型就要学会重新组合它们来理解日期格式；而如果切成["2024-01", "-15"]，模型看到的就是一个完全不同的结构。Tokenizer 对代码、数学公式、特殊字符的切分方式，都会影响模型在相关任务上的表现。

主流模型使用 BPE（Byte Pair Encoding）或其变体：算法从单个字符出发，反复合并最高频的字符对，最终形成一个词表——词表里每个条目就是一个 token，可以是单个字符、常见词片段，也可以是整个单词。词表大小通常在 3-10 万个 token 之间。

### 数据配比：能力分布的"配方"

这是整个数据工程阶段最关键的决策，也是最难公开讨论的部分——因为模型公司几乎不公开具体配比数据。

数据配比的逻辑是：**模型的能力分布，直接对应训练数据的领域分布**。代码数据占比高，写代码就强；中文语料薄，中文场景就弱；数学推理数据比例低，数学题就做得差。这不是模型设计的失误，而是训练数据配比的直接结果。

以 DeepSeek V4 为例。截至 2026 年 4 月，DeepSeek V4-Pro 的预训练使用了 33 万亿（33T）个 token，是上一代 V3（14.8T token）的 2 倍多。这种规模扩张本身是一个信号：在主流模型都已经把互联网高质量文本吃了一遍的情况下，继续扩规模意味着更多合成数据、更精细的配比策略、以及在特定领域数据上的定向增强。

数据配比对 PM 的实际影响，集中体现在一个场景：**模型版本升级**。模型公司在新版本里调整了数据配比——也许是为了增强代码能力，结果中文写作表现有了微妙退化；也许是加入了更多数学数据，却让模型在情感理解类任务上变得更生硬。这种变化不会有任何官方通知，它会悄悄地出现在你的产品里，等你的用户反馈才会发现。

这就是见因果链表「①数据工程·数据配比调整」这行所揭示的问题：**升级前跑回归测试不是谨慎，是标准动作**。合同里要求供应商在版本升级前提前告知，并写入"核心能力不回退"条款，是有实际技术依据的保护条款，不是可有可无的法务措辞。

## ② 预训练：能力从哪里来

数据准备好之后，进入预训练阶段。这是三个阶段里算力消耗最大、周期最长的一个，也是产出最难直接测量的一个——因为预训练直接产出的基座模型，既没有界面，也不会对话，只会在给定文字后继续生成文字。

但理解预训练，是理解"为什么模型会有这些能力"的根本所在。

### 自回归建模：一个朴素目标背后的惊人结论

预训练的目标极其简单：给模型大量文本，让它反复练习"预测下一个 token"。

给定"今天天气很"，预测下一个 token 是"好"还是"差"还是"热"。给定一段代码，预测下一行。给定一篇文章的前半部分，预测后续内容。就是这样，循环几万亿次。

这个目标为什么有效？直觉上很难理解：仅仅是预测下一个词，怎么会让模型学会推理、总结、翻译，乃至编写代码？

关键在于：**如果你要真正做好"预测下一个 token"，你就必须理解文本背后的世界**。

要准确预测"苹果公司的市值在 2024 年超过了______"，模型就必须建立起关于苹果公司、市值概念、2024 年大约发生了什么事情的内部表示。要准确预测一段代码的下一行，模型就必须理解这段代码在做什么。要准确预测一段对话的下一句，模型就必须建立对说话人意图、情绪、上下文的理解。

预测下一个 token 这件事，在足够大的规模和足够高质量的数据下，迫使模型学会了对世界的压缩表示——这就是能力的来源。

这一点有一个重要推论：**预训练阶段学到的能力，不是规则，而是统计结构**。模型"知道"地球围绕太阳转，不是因为它被编程了这条规则，而是因为在训练数据里，讨论地球和太阳关系的文本里，这个方向的表述压倒性地多于反方向。这意味着模型的"知识"本质上是训练语料的统计摘要——这解释了为什么模型会产生幻觉（统计上合理但事实上错误的输出），以及为什么对于训练数据覆盖极少的领域，模型表现会明显变差。

### Transformer 架构与注意力机制

现代大语言模型几乎全部基于 Transformer 架构。它的核心机制是**注意力机制（Attention Mechanism）**，理解这个机制，能解释很多产品层的现象。

注意力机制的作用：在处理一个 token 的时候，模型不是只看这个 token 本身，而是可以"注意"到序列中任意其他位置的 token，并根据相关程度分配不同的权重。

用一个例子建立直觉：

"银行坐落在河边，它的存款利率最近降低了。"

这句话里，"它"指代"银行"而不是"河"。注意力机制让模型能跨越"坐落在河边"这几个词，把"它"和"银行"正确地关联起来——并且在这里，"金融机构"含义的"银行"权重远高于"河岸"含义的"银行"，因为后面的"存款利率"强烈指向金融含义。

这个机制有两个对 PM 有实用价值的延伸：

**上下文窗口的物理基础**：注意力机制要求序列中的每一个 token，都去"关注"它前面所有的 token，并计算两两之间的相关权重。这意味着序列有多长，每个 token 就要做多少次计算——序列翻倍，总计算量翻四倍，呈平方级增长。上下文窗口的上限，本质上是这个计算成本的承受边界：不是工程师任意设定的，而是有真实的算力代价。2026 年主流模型的上下文窗口已经扩展到 1M token（Gemini 3.1 Pro、DeepSeek V4、Qwen3.7-Max 等），这背后是大量针对注意力计算的工程优化（稀疏注意力、滑动窗口注意力等），但"长上下文成本更高"这个基本规律没有改变。

**Lost in the Middle（信息遗忘中间段）**：注意力机制对序列开头和结尾的信息权重更高，中间段的信息在长上下文中容易被"遗忘"。这就是为什么当你给模型一个很长的 System Prompt 时，如果关键约束写在中间而不是开头或结尾，执行效果会明显变差。这不是模型不够"聪明"，而是注意力机制的结构性特征。

### Scaling Law 与涌现能力

缩放定律（Scaling Law）是预训练阶段最重要的经验规律：在数据量、模型参数量、计算量三个变量上同步扩展，模型性能会以可预测的方式提升。

这个规律有几个重要的含义：

**预测性**：模型公司可以在小规模实验上测出性能曲线，然后外推到大规模训练上预期的性能——这是训练千亿参数模型的底气来源，不是靠运气，而是靠规律。

**可量化的投入产出**：在 Scaling Law 成立的范围内，花多少算力、大约能得到什么质量的模型，是可以提前估算的。这也是为什么 2026 年前沿模型的算力投入规模如此巨大——Anthropic 为此支付的年化算力成本在 150 亿美元量级，这不是烧钱，是在 Scaling Law 框架下做有预期回报的投资。

**涌现能力**：Scaling Law 的一个重要现象是"涌现"——某些能力不是随着规模增长而线性增长的，而是在越过某个阈值后突然出现。少样本学习（Few-shot Learning）、思维链推理（Chain-of-Thought，见后续 PE 篇详述）、指令遵循，都是这样的涌现能力。这解释了为什么有时候从一个小模型换到一个大模型，某类任务的表现会出现跳跃式提升，而不是线性改善。

现在回到引子里的路线分歧：Ilya 说"预训练将终结"，指的是"用现有方式堆数据堆算力"这条路已经接近边界；Karpathy 和 DeepSeek 的行动则在说，**预训练还有空间，只是变量变了**——从"数据量 × 算力"转向"数据质量 × 架构效率 × 研究速度"。DeepSeek V4 把训练数据从 14.8T token 扩到 33T token，同时引入 Muon 优化器和稀疏注意力架构（DSA），是在用效率换质量，而不是暴力堆规模。

**这对 PM 的实际意义**：预训练路线的变化，会在 6-12 个月后以模型能力变化的形式传导到产品里。理解这件事，你就能提前预判——当某家供应商的下一个模型版本出来时，它的能力跃升点大概率在哪里，而不是等更新日志出来才被动应对。

### 基座模型的产出形态

预训练结束，产出的是**基座模型**。

它有几个关键特征：它博学但不会对话；它没有角色感，不知道自己是"助手"；它不知道拒绝，也没有安全意识；给它一段话，它就继续往下生成——语言流畅，但方向不受控。

经典的演示：你问基座模型"法国的首都是哪里"，它可能给你生成"这是一道常见的地理题，以下是本题的答案和解析……"——因为在它的训练数据里，这种问题后面跟着的往往是教辅内容，而不是"巴黎"这两个字。

让基座模型变成你能用的对话模型，是下一个阶段的任务。

## ③ 对齐训练：让模型变成"可用的"

预训练给了模型能力，对齐训练给了模型"形状"。

"形状"指的是：模型用什么方式回答问题，回答风格是什么，什么该说什么不该说，工具怎么调用，以及各种你在产品里感知到的行为模式——这些都是对齐训练写进权重的结果，而不是 Prompt 决定的。

理解这件事，是 PM 最值得投入的一个认知：**你在产品层看到的模型行为，有很大一部分不是"模型本来的样子"，而是对齐训练塑造出来的样子**。这意味着通过对齐可以改变模型，也意味着某些行为即使你写了 System Prompt，也无法根本改变——因为它们是权重级的，不是 Prompt 级的。

对齐训练包含三种主要方法：SFT、RLHF、DPO。其中 SFT 在前，偏好对齐在后——RLHF 和 DPO 是后一步的两种方案，DPO 是 RLHF 的简化替代，通常二选一使用。

### SFT：监督微调——教模型"怎么回答"

SFT（Supervised Fine-Tuning，监督微调）是对齐训练的第一步，做的事情最直接：准备大量高质量的"问题+标准回答"对，让模型在基座能力的基础上，学会以这种方式回答问题。

SFT 数据的质量和覆盖度，决定了模型的"基础行为模式"：

**指令遵循格式**：当你在 System Prompt 里要求"以 JSON 格式返回结果"，模型能不能稳定执行——这种能力不是天生的，是 SFT 阶段用大量"指令+符合格式的回答"训练出来的。

**对话轮次管理**：模型知道自己是"助手"，知道用户是"用户"，知道多轮对话里怎么维持上下文——这些基本的对话能力，是 SFT 阶段建立的。

**工具调用能力**：这是对 PM 理解产品最有价值的一点。模型"知道"有工具可以调用、"知道"何时应该调用、"知道"怎么生成正确格式的调用参数——这套能力不是模型公司在 System Prompt 里写了一段工具描述就有的，而是 SFT 阶段用大量"用户请求+工具描述+标准工具调用 JSON"数据训练进权重的。

这解释了为什么不同模型在工具调用上的表现差异这么大：有些模型在 SFT 阶段覆盖了丰富的工具调用场景，调用准确率高、格式稳定；有些模型的 SFT 数据里工具调用场景薄，在复杂工具组合场景下就容易出错。当你的工具调用场景复杂度提升后，某些模型会遇到准确率天花板——这时候不应该继续优化 Prompt，而应该先判断这是否是 SFT 覆盖度不足导致的模型天花板（见因果链表「③对齐训练·SFT 工具调用覆盖度」一行）。

### RLHF：基于人类反馈的强化学习——教模型"什么是好回答"

SFT 解决了"模型会怎么回答"，RLHF（Reinforcement Learning from Human Feedback）解决的是更细腻的问题：在多种可能的回答里，哪种是人类更偏好的？

RLHF 的基本流程：

1. 让模型对同一个问题生成多个不同回答
2. 人工标注员对这些回答进行排序（A 优于 B，B 优于 C）
3. 用这些偏好数据训练一个"奖励模型"——一个学会了"人类觉得什么是好回答"的模型
4. 用这个奖励模型来引导主模型：生成被奖励模型评分高的回答，模型就被正向强化；生成评分低的回答，就被负向强化

这个过程反复迭代，最终让主模型的输出风格向人类偏好收敛。

RLHF 塑造了模型最明显的"个性"特征：回答的风格（正式还是随意、简洁还是详细、保守还是大胆）、对争议话题的处理方式、拒绝敏感请求的倾向和措辞——这些都是 RLHF 阶段人类标注员的集体偏好写进权重的结果。

这就引出了**权重级偏好**这个概念，也是见因果链表「③对齐训练·RLHF 偏好倾向」这行所说的核心问题：

**有些模型行为是写进权重里的，不是 Prompt 可以完全覆盖的。**

举几个具体例子：你在 System Prompt 里要求模型"回答要非常简洁，不超过两句话"，但模型总是倾向于给出一个稍长的解释——因为它的 RLHF 数据里，更完整的回答往往得到了更高的人类评分，这个偏好已经写进权重，一条 Prompt 指令在统计上会让它变短，但无法完全消除这种倾向。

又比如：你希望模型在某些场景下表现得更"直接"，少一些铺垫和礼貌用语——但如果这个模型的 RLHF 数据偏向了礼貌风格，你会发现无论怎么写 System Prompt，都很难让它真正"硬起来"。

识别这条边界的实用方法：如果你对同一个行为倾向做了很多次 Prompt 调整，每次都有一点点效果但效果有上限，很可能你碰到的是 RLHF 权重级行为，而不是 Prompt 写法问题。遇到这种情况，继续死磕 Prompt 是在浪费资源——应该考虑换一个和你产品需求风格更匹配的模型，或者评估是否有必要做 Fine-tuning。

### DPO：直接偏好优化——RLHF 的工程简化

DPO（Direct Preference Optimization，直接偏好优化）是 RLHF 的简化替代方案，在 2024 年后成为主流。

RLHF 的痛点：需要单独训练一个奖励模型，然后用强化学习算法（PPO）来优化主模型——整个流程复杂、不稳定、对超参数敏感，训练成本很高。

DPO 的思路：绕过奖励模型和 PPO，直接用"好回答 vs 坏回答"的配对数据来优化主模型——数学上可以证明，这在特定条件下等价于 RLHF 的目标，但训练过程简单得多，稳定性也更好。

对 PM 来说，DPO 最重要的含义不是技术细节，而是一个行业信号：**对齐训练的工程成本在下降**。这意味着更多模型公司能以更低成本完成对齐迭代——模型公司有时会在不重训基座的情况下只更新对齐部分，发布行为改善的新版本。你现在选的模型，3-6 个月内可能就通过一次对齐版本更新，修掉某个你目前忍受着的风格或行为问题。这是选型时值得纳入考量的变量：这家公司的对齐迭代能力怎么样？

### Constitutional AI：用原则做对齐

Constitutional AI（CAI，宪法 AI）是 Anthropic 提出和使用的对齐方法。基本思路是：给模型一组原则（"宪法"），让模型用这些原则对自己的输出进行批判和修正，最终用自我批判的结果来做对齐训练——减少了对大量人工标注的依赖。

对 PM 最有实际意义的是 CAI 的一个产物：**安全红线是写进权重的**。

Anthropic 的模型（和其他经过类似安全对齐的模型）有一些行为边界，是用 CAI、RLHF 等方法在训练阶段建立的，它们存在于权重层面，而不是规则层面。这意味着：

- **你可以依赖它**：不需要在 System Prompt 里写大量"禁止生成 XX 内容"的规则，因为模型本身对这些内容有抵触。这是 Guardrails 设计里，训练层提供的第一道防线。
- **你不能绕过它**：如果你的产品场景需要模型做某些它的安全训练认为有问题的事情——比如某些医疗、法律、安全研究场景——System Prompt 和 Prompt 技巧在这里作用有限，你需要和模型供应商谈判获取特定权限，或者选择安全限制相对宽松的模型。
- **这是选型变量**：不同模型的安全对齐强度和边界不同——这不是"哪个更安全"的价值判断，而是"谁的边界和你的场景更匹配"的选型判断。

## 写在最后：PM 决策框架

这场 Karpathy vs Ilya 的争论，你不需要站队，但需要看懂机制。三个阶段讲完，机制已经在这里了——数据怎么决定能力分布，预训练怎么划定天花板，对齐训练怎么把偏好写进权重。

把它收进三个问题，放在手边，遇到事的时候取用：

**模型表现不符合预期时：是训练层造成的，还是应用层造成的？**

先做一个快速归因：换一个强得多的模型，问题是否消失？如果消失，是模型能力天花板；如果依然存在，问题在应用层的 Prompt、RAG 或配置。如果只是换了同级别的不同模型，问题程度有变化，方向大概率是对齐训练的风格差异（见因果链表「③对齐训练·RLHF 偏好倾向」）。

**反复调 Prompt 有效果但有上限时：这个行为是权重级的，还是 Prompt 级的？**

碰到这个信号，大概率已经在和权重级偏好较劲了。继续死磕 Prompt 边际递减，值得停下来评估是否换模型或做 Fine-tuning。

**模型版本升级前：有没有一套测试集可以跑？**

版本升级是训练层变化最集中的传导时刻——①数据工程·数据配比可能调整，②预训练·能力分布可能变化，③对齐训练·行为倾向可能漂移。有 Golden Set 就能在用户反馈之前发现问题；没有的话，只能等着被动响应。
